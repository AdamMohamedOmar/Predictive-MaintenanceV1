# Real-fault collection protocol — Skoda Roomster 2007 1.6L

**Audience:** Adam and Ahmed, executing the protocol without a mechanic in the
loop.
**Purpose:** Collect real OBD-II recordings of induced faults on the Skoda so
the detection pipeline (classifier, IsolationForest, PID forecaster) can be
evaluated against a signal the team did not synthesise.
**Headline metric:** vacuum-leak recall ≥ 0.60 on the windows during the
fault interval (see §10 for the exact computation).

This document is the v1.0 protocol; updates land via PR to this file with a
change-log entry at the bottom.

---

## 1. Why this exists

The classifier, anomaly detector, and PID forecaster all train against
windows whose fault labels and severity values were generated by the
deterministic injector in `src/injection/fault_injector.py`. They have
never been evaluated against a fault the team did not write the formula
for. The metrics published in `results/*.json` are therefore self-
consistency floors (see project root `README.md`, "Headline numbers"),
not real-fault detection results.

This protocol gathers the data that closes that gap. It describes safe,
reversible inductions of two real fault conditions on the Skoda (vacuum
leak, ECT bias), one substitute (baseline-drift robustness probe in place
of TPS drift, which cannot be safely induced on a healthy vehicle), and
documents the lean-condition overlap that lets us evaluate fuel-system
detection without inducing a real fuel-system fault.

---

## 2. Safety doctrine — applies to every recording

These rules are mandatory. If any one of them cannot be satisfied for a
given induction, **skip that induction**. Do not proceed and re-evaluate
later — re-plan first.

1. **Park the car, set the handbrake, and turn the engine off** before
   inducing any fault. Inductions happen at rest.
2. **No driving above 30 km/h** during the vacuum-leak induction. Lean
   conditions damage the catalytic converter over time; the protocol's
   time windows are calibrated around that risk.
3. **Have an OBD-II scan tool capable of clearing DTCs on hand** before
   inducing the ECT bias. If the ECU sets a permanent code the dashboard
   alone cannot clear it.
4. **Abort on any unexpected DTC.** The expected DTCs are listed per
   induction. Anything else → restore the vehicle to baseline, do not
   drive home with the modification in place.
5. **Two people on every recording.** One drives, one watches the
   dashboard / scan tool / clock. Solo recordings are not allowed.
6. **Daylight, dry conditions, fuel level ≥ ¼ tank.** No night-time
   inductions, no rain (electrical work on a wet engine bay), no fuel
   sloshing artefacts.
7. **Smell test before driving.** After installing any modification, idle
   the engine for 60 s and confirm no fuel smell, no smoke, no warning
   lamp besides the one(s) you expect.
8. **Restore-before-leaving.** Every induction ends with the vehicle
   physically returned to its pre-induction state. The "post" recording
   interval is for confirming this — if the post interval doesn't return
   to healthy readings, do not consider the session complete.

Abort criteria summary:

| Symptom | Action |
|---|---|
| Engine stall at idle after induction | Restore immediately; do not drive |
| MIL on with an unexpected DTC | Restore, clear DTC, do not drive home with the modification |
| Fuel smell, smoke, or visible leak | Restore immediately, ventilate, do not drive |
| Limp mode (RPM capped, throttle unresponsive) | Restore immediately, ignition cycle, re-verify before driving |
| Coolant temp >110 °C during ECT-bias test | Restore, idle, do not drive |

## 4. Recording protocol — applies to every induction

### 4.1 Intervals

Every recording is three concatenated intervals:

| Interval | Duration | What happens |
|---|---|---|
| `pre` | 5 minutes (300 s) | Vehicle running with **no modification**. Idle and gentle driving in a quiet parking lot. Establishes a clean healthy baseline on the day of the recording. |
| `fault` | per-induction (15–20 min) | Modification in place. Drive per the specific protocol below. |
| `post` | 5 minutes (300 s) | Modification removed. Confirm vehicle returns to healthy readings before ending the recording. |

Total session is therefore 25–30 minutes of continuous recording at 1 Hz
(1500–1800 rows). This is roughly the same length as a `live*.csv` session
in `data/raw/carOBD/`.

### 4.2 File format

The dashboard's CSV log writes clean-column format (e.g. `ENGINE_RPM`,
not `ENGINE_RPM ()`). That matches what `CsvStreamer._load_csv` and
`src/eval/real_fault_eval.py:_read_csv` accept — both fall back to plain
`pd.read_csv` when the raw-carOBD column rename map doesn't apply.

Filename: `skoda_<faulttype>_<YYYYMMDD>_<runN>.csv`.

Example: `skoda_vacuumleak_20260605_run1.csv`.

### 4.3 Sibling metadata JSON

Every CSV ships with a `.json` sibling carrying the run's context:

```json
{
  "vehicle": "skoda_roomster_2007_1.6L",
  "fault_type": "vacuum_leak",
  "induction_method": "PCV-line cap, 4 mm rubber cap with hose clamp",
  "materials": ["4 mm rubber cap", "4 mm hose clamp", "4.7 kΩ resistor (unused this run)"],
  "mods_in_place_from_s": 300,
  "mods_removed_at_s": 1200,
  "operator_drive": "Adam",
  "operator_log":   "Ahmed",
  "weather": "23 °C, dry, no wind",
  "fuel_grade": "92 RON",
  "fuel_level_at_start": "5/8",
  "session_id": "skoda_vacuumleak_20260605_run1",
  "notes": "Engine set P0171 at t=920s; dashboard logged it; not cleared until post-interval end."
}
```

The eval harness (`src/eval/real_fault_eval.py`) uses
`mods_in_place_from_s` and `mods_removed_at_s` to slice the recording
into `pre / fault / post` intervals when computing recall. Everything
else is for the paper's data-collection section.

### 4.4 Storage

CSVs and JSONs land in `data/real_faults/skoda/`. The directory is
gitignored; files are committed selectively after review (file size,
PII check on the JSON metadata — strip operator names if publishing).

## 5. Fault 1 — vacuum leak (**primary**, safe-reversible)

This is the centrepiece. Of all four fault classes the team trained
against, the vacuum leak is the only one we can safely induce, fully
reverse, and observe with the existing dashboard. The headline real-
fault metric in §10 is computed against this induction.

### 5.1 Mechanical premise

A small unmetered air leak downstream of the MAF / MAP sensor injects
extra air into the intake the ECU does not see. The fuel-air mixture
goes lean, the O₂ sensor reports lean exhaust, and the short-term fuel
trim (STFT) climbs positive within ~30 s. Over the following minutes
the long-term fuel trim (LTFT) integrates the STFT offset and shifts
positive too. The ECU may eventually set diagnostic trouble code P0171
(System Too Lean, Bank 1) after 10–20 minutes if the leak is severe
enough.

### 5.2 Materials

- One 4 mm inner-diameter rubber cap (any auto parts store; ~20 EGP)
- One 4 mm hose clamp or a heavy-duty zip tie
- Latex gloves
- Flashlight

### 5.3 Target hose

The smallest accessible vacuum hose on the Skoda 1.6L intake — the
brake-booster check-valve vent line or a PCV vent line are typical
candidates. Pick the smallest because a small leak is recoverable; a
large leak (e.g. disconnecting the main intake hose) will stall the
engine and is out of scope for this protocol.

**Do not cut, drill, or otherwise damage any part of the engine bay.**
The induction is "disconnect and cap one end of a small hose."

### 5.4 Procedure

| Step | Action | Verify |
|---|---|---|
| 1 | Park, handbrake, engine off, key out. | Vehicle still. |
| 2 | Connect ELM327, launch dashboard, select "Live vehicle" source, press Play. | Dashboard shows live PIDs. |
| 3 | Re-start engine. Begin the 5-minute `pre` interval — idle and very gentle driving in the parking lot. | Dashboard shows healthy classifier label for ≥ 4 of the last 5 minutes. |
| 4 | Park, engine **off** (this is important — never modify a running vacuum system). Log `mods_in_place_from_s` = current dashboard elapsed_s. | Engine stopped. |
| 5 | Identify the target hose. Disconnect it at one end (typically the engine-side fitting). Cap the open end with the rubber cap. Secure with the hose clamp or zip tie. Leave the other end disconnected — air will now leak into the intake through that opening. | Hose visually capped; no kinks; cap will not fall off at engine vibration. |
| 6 | Re-start the engine. Idle 60 s. | No stall. No fuel smell. No smoke. RPM stable within ±50 of normal idle. |
| 7 | Begin the `fault` interval — 15 minutes of gentle driving (city speeds ≤ 30 km/h, no aggressive acceleration). Operator-log notes any DTC that appears on the dashboard, including the elapsed_s. | STFT climbs positive within 30 s; LTFT begins drifting positive within ~3 min. |
| 8 | At fault+15 min, park, handbrake, engine **off**. Reverse step 5: remove cap, reconnect hose, secure with original fitting. Log `mods_removed_at_s`. | Hose visually re-seated; cap removed from vehicle. |
| 9 | Re-start engine. Begin the 5-minute `post` interval — gentle driving. | STFT and LTFT return toward `pre` values within 3 minutes. |
| 10 | Park, engine off, stop the recording. Save the CSV + JSON. | File written to `data/real_faults/skoda/`. |

### 5.5 Expected DTC behaviour

A small leak typically does not set a code in 15 minutes. A larger leak
may set **P0171 — System Too Lean, Bank 1**. This is the *expected* DTC
for this induction. If it appears, log it but do not clear it during
the run. Clear it during the post interval or after parking with the
external scan tool.

If a different DTC appears (e.g. P0300 misfire, P0420 catalyst
efficiency) — **abort.** Restore the vehicle, drive home only after the
scan tool reports the live readings are within range.

### 5.6 Expected OBD-II signature on the dashboard

- `SHORT_TERM_FUEL_TRIM_BANK_1`: rises from ~0 % to ~+8 to +15 % within 30 s
- `LONG_TERM_FUEL_TRIM_BANK_1`: rises slowly from baseline to ~+5 to +12 % over 5–10 min
- `INTAKE_MANIFOLD_PRESSURE`: rises slightly for a given throttle position
- `ENGINE_LOAD`: rises slightly for a given throttle position
- Classifier label: most post-bias windows flagged as `air_system` or `fuel_system`

## 6. Fault 2 — coolant temperature sensor bias (**secondary**, gated)

This induction is more invasive than the vacuum leak. It must not be
attempted unless **all three** preconditions in §6.4 are satisfied.

### 6.1 Mechanical premise

The engine control unit reads coolant temperature off a thermistor — a
resistor whose resistance falls as temperature rises. By splicing a
fixed 4.7 kΩ resistor in series with the sensor's signal wire we raise
the apparent resistance, biasing the reading toward "cold". The ECU
believes the engine never reached operating temperature, retards
ignition timing, and enriches the fuel mixture slightly.

### 6.2 Materials

- One 4.7 kΩ ¼ W resistor (any electronics shop; ~5 EGP)
- Two insulation-displacement splice connectors sized for the Skoda ECT
  signal wire gauge (typically 0.5 mm² / 20 AWG)
- Wire strippers (optional — splice connectors usually don't need them)
- Wiring diagram for the Skoda 1.6L ECT sensor pinout (see §6.4
  precondition 1)
- A known-bad junkyard ECT sensor for the bench test (see §6.4
  precondition 3)
- Latex gloves, flashlight

### 6.3 Procedure (only after §6.4 preconditions met)

| Step | Action | Verify |
|---|---|---|
| 1 | Park, handbrake, engine **off and cool** (coolant below 40 °C). | Touch the upper radiator hose — must be cool to the touch. |
| 2 | Connect ELM327, launch dashboard, record `pre` interval (5 min idle + gentle drive). | Healthy classifier label for ≥ 4 of 5 min. |
| 3 | Park, engine **off**. Locate the ECT signal wire per the wiring diagram. Install splice connectors with the resistor in series, *without* cutting the wire (insulation-displacement splice taps the existing conductor). | Resistor installed in series with the signal wire only. |
| 4 | Re-start engine. Verify ECT reading on the dashboard drops by 15–25 °C compared to the ambient-corrected expected value. **If the reading drops to <0 °C or above the previously-observed maximum** — abort, the ECU may interpret the value as a sensor fault and enter limp mode. | Dashboard COOLANT_TEMPERATURE 15–25 °C below true coolant temp. |
| 5 | Log `mods_in_place_from_s`. Begin 20-minute `fault` interval — idle and gentle driving. | TIMING_ADVANCE retards 3–5°; fuel trims drift slightly rich (negative). |
| 6 | At fault+20 min, park, engine **off**. Remove splice connectors and resistor. Restore the original signal wire. | Wire visually intact; splice connectors removed; nothing left in the engine bay. |
| 7 | Re-start engine. Begin 5-minute `post` interval. | ECT reading returns to true value within 60 s; TIMING_ADVANCE returns to baseline. |
| 8 | Park, engine off, stop recording. Clear any DTC with the external scan tool. | Healthy classifier label restored in post interval. |

### 6.4 Preconditions — **all three required**

1. **A Skoda Roomster 2007 1.6L ECT signal-wire diagram in hand.**
   If you cannot find a wiring diagram from a Haynes manual, the ELSA
   workshop database, or a verified online source, **skip this fault.**
   Do not guess at wiring on a running car.
2. **An OBD-II scan tool that can clear DTCs.** The ECU is likely to
   set **P0118 (ECT Circuit High Input)** when the apparent resistance
   exceeds the expected range. P0118 is the *expected* DTC. Other DTCs
   are abort criteria.
3. **A successful bench test on a junkyard ECT sensor.** Before doing
   this on the Skoda, obtain a junkyard ECT sensor (any vehicle), wire
   it up on a bench with a known-good ohmmeter and a hairdryer for
   heat, and confirm the resistor splice does what §6.1 says it does.
   This catches wiring-direction mistakes before they reach the car.

### 6.5 Expected limp-mode risk

If the ECU rejects the biased reading as out-of-range, it can enter
limp mode (reduced power, fixed throttle map, MIL on). Limp mode is
recoverable by removing the resistor and cycling the ignition. Drive
the vehicle only after the ECT reading is back in range and the MIL is
cleared. **This is the primary reason §6.4 precondition 3 (bench test)
exists** — to confirm the resistor value lands inside the ECU's
acceptable range before the splice goes on the car.

### 6.6 Expected OBD-II signature

- `COOLANT_TEMPERATURE`: drops 15–25 °C below true value within 30 s of
  splice; stays low for the whole `fault` interval
- `TIMING_ADVANCE`: retards 3–5° below expected for the actual temperature
- `SHORT_TERM_FUEL_TRIM_BANK_1`: drifts slightly negative (rich)
- Classifier label: post-bias windows flagged as `coolant_temp_sensor`
- Possible DTC: **P0118** (expected; do not clear during the run)

## 7. Fault 3 — substituted as a baseline-drift robustness probe

Real throttle position sensor drift cannot be safely induced on a
healthy Skoda. A TPS fault is a hardware-degradation pattern — the
sensor's resistive track wears unevenly over years. Replicating it
requires either a known-bad TPS module (rarely available) or hardware
modification of the existing module (irreversible, expensive).

Instead, we use the day's Skoda recording as input to a **baseline-drift
robustness probe** of the detection algorithm. This is reported in the
paper's `Limitations` section, **not** the headline results table.

### 7.1 Procedure

1. Record 30 minutes of normal Skoda driving with no modification.
   Filename: `skoda_baselinedrift_<YYYYMMDD>_run1.csv`.
2. Feed the recording through `src/eval/real_fault_eval.py` twice:
   first with the Skoda's true normalizer baseline (captured via
   `scripts/live_baseline_capture.py`), then with the baseline
   intentionally shifted by +0.30 on the throttle-to-pedal ratio.
3. The detection algorithm should flag the shifted-baseline run as
   `throttle_position_sensor` more often than the true-baseline run.
   That demonstrates algorithm sensitivity to baseline drift, but it
   does **not** demonstrate real-fault detection.

### 7.2 Reporting

In the paper, this experiment is described as a "robustness probe":
the algorithm's behaviour when the per-vehicle normalizer baseline is
inaccurate. It belongs in the threats-to-validity / limitations
discussion, alongside the synthetic self-consistency floor.

The headline metric in §10 is **not** computed against this run.

## 8. Fault 4 — fuel system, declared limitation

Real fuel-system faults (clogged injector, low rail pressure, failing
pump) cannot be safely induced on a healthy fuel system. Inducing them
requires modifying the fuel rail, the injectors, or the pump itself —
all of which are expensive, irreversible, and create real risk of fire
or stranded vehicle.

We declare fuel-system real-fault detection as **future work** in the
paper's limitations section.

The vacuum-leak induction in §5 produces a sustained lean condition
that overlaps in OBD-II signature with a fuel-system fault (both push
STFT and LTFT positive). For paper reporting, **"lean condition
detected" (classifier label ∈ {air_system, fuel_system})** is scored as
success on the vacuum-leak induction. The confusion between the two
classes on the vacuum-leak run is itself a useful finding for the
paper's discussion of fault-class separability with the existing
13-PID feature set.

## 9. After-action — at the laptop

When a recording is in hand:

1. Run the harness:

   ```
   python -m scripts.eval_real_fault data/real_faults/skoda/skoda_vacuumleak_20260605_run1.csv
   ```

   Output JSON lands at `results/real_fault_eval/skoda_vacuumleak_20260605_run1_v1.json`.

2. Open that JSON and read the `summary.label_counts` and the
   per-window list. Manually classify each window as `pre`, `fault`, or
   `post` using `mods_in_place_from_s` and `mods_removed_at_s` from the
   sibling metadata JSON.

3. Compute vacuum-leak recall (see §10).

4. File the run's CSV + JSON + computed recall in `results/real_fault_eval/`
   and add a row to `results/real_fault_eval/runs_index.md` (created on
   first commit; one row per run, columns: filename, date, induction,
   recall, notes).

## 10. Headline metric — vacuum-leak recall

**Definition.** Of the windows whose `elapsed_s` falls inside the
`fault` interval (`mods_in_place_from_s ≤ elapsed_s ≤ mods_removed_at_s`),
the fraction flagged as **either** `fuel_system` **or** `air_system` by
the classifier **or** scored above the anomaly-detector threshold of
0.85.

```
recall_vacuum_leak = |{windows where label ∈ {fuel_system, air_system}} ∪ {windows where anomaly_score ≥ 0.85}|
                     ────────────────────────────────────────────────────────────────────────────────────────
                     |{windows where mods_in_place_from_s ≤ elapsed_s ≤ mods_removed_at_s}|
```

**Pass criterion: recall ≥ 0.60.**

If the project ships with a recall of 0.60 or higher on the vacuum-leak
induction, the paper's headline contribution is "a fault-detection
pipeline that detects a real vacuum leak with 60 %+ recall, evaluated
against data the team did not synthesise". If recall is below 0.60,
the paper's headline reframes to "a fault-detection pipeline plus
collection protocol; recall validation is future work."

That reframing is honest — and was scoped explicitly in §11 invariant
#7 of `docs/CHARTER.md` (the v1.1.1 interim charter amendment).

## 11. What this protocol does **not** cover

- **Cold-start regime classification.** That class is a normal operating
  state, not a fault; testing it just means recording a cold engine
  start, which is trivial and doesn't need a protocol.
- **Misfire detection.** Out of scope per CHARTER §4.2 — 1 Hz OBD-II
  cannot resolve per-cylinder combustion events.
- **Naturally degraded sensors.** The whole protocol is about *induced*
  faults. Run-to-failure data collection is explicitly future work per
  CHARTER §11 invariant #3.
